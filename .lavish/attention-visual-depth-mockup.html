Vector transform
Q / K / V projection: computing q₀ for "the"

A single embedding vector, multiplied by one row of a weight matrix, produces one output number. Zooming from the full matrix down to one row down to one multiply-and-sum is the same zoom every projection in this page uses.

01: STORAGE
The full data at rest

Every token in this example starts as a 4-number vector called an embedding. The vector below is the embedding for the token "the" — these 4 numbers were picked by hand for this toy example; a real trained model would have learned them from data. Below it is W_Q, a 4×4 matrix of weights. It's one of three such matrices attention learns (the others are W_K and W_V) — this one's specific job is turning any token's embedding into that token's query vector.

x "the" — the raw embedding (d=4)
W_Q — 4×4 learned weight matrix
1 vector + 1 matrix, both stored in full
02: SLICE
Pull out row 0 of W_Q

We're computing just one number: the first entry of the query vector, q₀. Producing q₀ needs only one of W_Q's four rows — row 0, highlighted below. The other three rows (dimmed) would each produce a different output number (q₁, q₂, q₃) and play no part here. x is unchanged from stage 1, placed here so it lines up directly against the row it's about to combine with.

x "the" (same as stage 1)
W_Q, row 0 only
row i of W_Q always produces output dimension i
03: TRANSFORM
Multiply position by position, then sum

Pair up x and the row from stage 2 by position: x's 1st number with the row's 1st, x's 2nd with the row's 2nd, and so on. Multiply each pair, then add all four products together. That sum is q₀ — nothing more happens to it.

↓ add the 4 products
04: CONCEPT
What this means
◎

Each output dimension is its own learned blend of every input dimension — nothing in x is ignored, and nothing in the output comes from just one input number. Row 0 of W_Q, in effect, asks: "how much does each of this token's 4 embedding features contribute to query-dimension 0?" Rows 1-3 ask the same question for dimensions 1, 2, and 3. Repeating this whole stage for every row of W_Q, then separately for W_K and W_V, is exactly how the full q, k, and v vectors get built.

Matrix formation
QKᵀ scores: computing score["the","the"]

Two full matrices of vectors, one row from each pulled out and compared, landing in one cell of a brand new matrix. The full QKᵀ step repeats this nine times, once per query/key pair.

01: STORAGE
Every query, every key

The previous step (Q/K/V projection) already produced a query vector and a key vector for every token, not just "the". Q below stacks all three query vectors, one row per token; K stacks all three key vectors the same way. Both are just the outputs of that earlier step, collected together — nothing new is computed on this stage.

Q — one row per token (3×4)
K — one row per token (3×4)
3 queries × 3 keys = 9 comparisons still to come
02: SLICE
One query, one key

To fill in exactly one cell of the score grid — the cell where query "the" meets key "the", i.e. row 0, column 0 — we only need one row from Q and one row from K, both highlighted above. Every other row of Q or K belongs to a different cell of the grid and isn't used for this particular number.

q "the" — row 0 of Q
k "the" — row 0 of K
this pair lands in score grid cell [0,0]
03: TRANSFORM
Multiply position by position, then sum

Same operation as the projection step's transform stage, just applied to two vectors that both came from projections rather than a vector and a matrix row: multiply q and k position by position, then add the four products. This total is called a dot product, and it's the standard way to measure how "aligned" two vectors are.

↓ add the 4 products
04: CONCEPT
What this means
▦

A large dot product means q and k point in a similar direction — loosely, "what this token is looking for" closely matches "what that token offers." Every cell of the grid below is the result of this exact same multiply-then-sum, just for a different query/key pair. The highlighted cell is the one you just watched get computed; the other eight were produced by repeating stages 2-3 for every other pairing.
